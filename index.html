End-to-End Workflow

The complete
patient journey

Eight steps from first tap to continuous care loop — each step shows who acts (user, agent, or system) and exactly what gets produced.

🗂️
Step 1 — Entry & Intent Capture
Triage & Scheduling Agent
User action: Opens NaviMed, selects visit type (new issue / follow-up / existing concern).

Agent action: Triage & Scheduling Agent captures structured intent — visit type, basic patient info, and reason for visit.

Output: Structured visit request created and published to event bus.
Example: User selects "New Issue" → types "chest tightness for 2 days" → Agent creates: { visit_type: "new_issue", reason: "chest tightness", duration: "2d" }
🚦
Step 2 — Intelligent Triage
Triage & Scheduling Agent
User action: Provides brief reason for visit.

Agent action: Evaluates urgency and context to determine appropriate care pathway — Emergency escalation, Telehealth, or In-person visit.

Output: Recommended care modality locked. High-urgency cases escalate immediately; routine cases flow to scheduling.
Example: "Chest tightness + shortness of breath × 2 days" → Agent flags as elevated → outputs: { modality: "in_person", urgency: "high", route: "cardiology" }
📅
Step 3 — Smart Scheduling
Triage & Scheduling Agent
User action: Selects preferred time or accepts the agent's recommendation.

Agent action: Retrieves available slots from Scheduling System API and suggests optimal options ranked by proximity, continuity, and availability rules.

Output: Appointment confirmed with modality defined. Confirmation sent to user. AppointmentConfirmed event fires — triggers Structuring Agent.
Example: "Dr. Patel (your PCP) has openings tomorrow at 9 AM and 2 PM. Based on your location, the 9 AM slot is 18 min away. Book it?"
📝
Step 4 — Pre-Visit Symptom Input
Structuring Agent
User action: Enters symptoms, concerns, and goals — either immediately after booking or via a reminder before the visit.

Agent action: Structuring Agent processes free-text input through LLM and organizes it into a structured clinical summary with primary concern, duration, key symptoms, and patient priorities.

Output: AI-generated provider-ready summary: { primary_concern, duration, symptoms[], patient_priorities[], flagged_risks[] }
Example summary: "Primary: chest tightness × 2 days. BP trending up (last 3 readings). Patient question flagged: Lisinopril side effects."
🏥
Step 5 — Provider Handoff
Structuring Agent
User action: None — this step happens automatically.

Agent action: Structuring Agent delivers the structured summary directly into the provider's workflow via EHR API. Publishes ProviderSummaryReady event.

Output: Provider receives structured pre-visit context on their dashboard and reviews before entering the room.
Provider sees: Marcus, 52M — Chief complaint: chest tightness × 2d — Elevated BP trend — Question: Lisinopril SE — No ER since last visit. (Provider acknowledges to unlock for clinical use.)
👨‍⚕️
Step 6 — Visit Execution
No Agent Active
User action: Patient attends the visit (telehealth or in-person).

System role: Supports passively. Provider uses the AI summary as a starting point for the interaction. No agent drives the visit.

Output: More efficient, focused visit with reduced redundancy. EHR records the encounter and fires VisitCompleted when the encounter closes — activating the Engagement Agent.
What changes: Provider doesn't need to re-ask "what brings you in today" — they open with "I see you've had chest tightness for two days and had a question about Lisinopril. Let's start there."
📬
Step 7 — Follow-Up & Engagement
Engagement Agent
User action: Receives follow-up messages and may respond or schedule next visit.

Agent action: Sends post-visit reminder, dispatches experience survey, and initializes follow-up window (6–12 months). Monitors provider schedule for matching slots.

Output: Follow-up actions tracked: reminder_sent, survey_dispatched, follow_up_window_started, slot_monitored
Patient receives: "Thanks for your visit! How did it go? [1-tap survey] → 6 months later: Dr. Patel just opened October slots — book your follow-up now."
🔁
Step 8 — Loop Back to Care
Engagement → Triage Agent
User action: Schedules follow-up or new visit via the portal.

Agent action: Engagement Agent fires CareLoopReEntry event with updated patient context. Triage & Scheduling Agent re-initializes the full flow from Step 1 — now with richer history.

Output: Continuous care loop maintained. Patient context (survey scores, visit history, medication changes) carried forward into the next triage cycle.
Loop payload includes: { context: "follow_up_6mo", last_visit: "2025-10-02", survey_score: 4.8, re_entry_reason: "slot_opened" }

AI Agent Layer

Three agents,
one care loop

NaviMed uses three specialized agents that hand off to each other via structured payloads on the event bus — each triggered by a specific user action or system event, each producing a defined output.

How the Agents Talk to Each Other

② Structuring Agent
Steps 4–5 of patient journey

→

③ Engagement Agent
Steps 7–8 of patient journey
Triggered by: VisitCompleted event fires from EHR
↙ Engagement Agent re-initiates Triage & Scheduling Agent when patient responds or schedules follow-up ↗
🗂️
Triage & Scheduling Agent
Steps 1 – 3
Captures visit intent, evaluates urgency, and orchestrates appointment booking across available care modalities.
⚡ What Triggers It
• User selects visit type (new issue / follow-up / existing concern)
• User submits brief reason for visit (free text)
• Engagement Agent fires re-entry event after follow-up confirmation
🔁 How It Communicates
Publishes VisitRequestCreated to event bus with structured payload → Scheduling System responds with available slots → confirmation published as AppointmentConfirmed, which triggers Structuring Agent.
📤 Output
Structured visit request: visit_type, urgency_level, care_modality (Emergency / Telehealth / In-person), confirmed_slot, provider_id, calendar_hook
Confirmation sent to patient. Modality decision (Emergency escalation, Telehealth, or In-person) is locked and passed downstream.
📋
Structuring Agent
Steps 4 – 5
Processes free-text symptom input into a structured clinical summary and delivers it to the provider workflow before the visit begins.
⚡ What Triggers It
• AppointmentConfirmed event from Triage Agent
• Patient submits symptom form (immediately or via reminder)
• Pre-visit reminder fires if form not yet completed
🔁 How It Communicates
Receives PatientSymptomInput payload → LLM processes free text → structured summary written to EHR API → publishes ProviderSummaryReady event. Provider dashboard surfaces summary for review.
📤 Output
AI-generated provider-ready summary: primary_concern, symptom_duration, key_symptoms[ ], patient_priorities[ ], flagged_risks[ ]
Example summary field: "Primary: chest tightness × 2 days. BP trending up. Patient question: Lisinopril side effects."
Provider must review and acknowledge before summary is used clinically.
📬
Engagement Agent
Steps 7 – 8
Manages all post-visit communication and re-engages the patient back into the care loop — handing off to Triage & Scheduling when they're ready to book again.
⚡ What Triggers It
• VisitCompleted event fires from EHR
• Patient responds to a follow-up message
• 6–12 month follow-up window timer elapses
• Provider schedule opens a matching slot
🔁 How It Communicates
Publishes PostVisitReminderSent, SurveyRequested, FollowUpPromptSent to event bus. On patient re-engagement, fires CareLoopReEntry → Triage & Scheduling Agent re-initializes with updated patient context.
📤 Output
Follow-up actions: reminder_sent, survey_dispatched, survey_response, follow_up_window_status, next_slot_notified, follow_up_booked
When patient responds or books, passes updated context payload back to Triage Agent — continuous care loop maintained.
👤
Step 6 — Visit Execution (No Agent Active)
The system supports passively during the actual visit. The provider uses the AI-generated summary as their starting point — no agent is driving the interaction. The EHR records the outcome and fires the VisitCompleted event when the encounter closes, which re-activates the Engagement Agent.

System Architecture

RAG-based, event-driven
platform layers

A modular architecture that separates clinical-risk logic from UI, supports async event-driven execution at scale, and makes policy changes independent of agent rewrites.

Patient App
Web / Mobile interface — symptom intake, booking, check-in, reminder center, survey, follow-up scheduling.
React + Vite Tailwind CSS Mobile-first QR check-in
API Gateway / BFF
Auth & session handling, request orchestration, client-specific response shaping for mobile vs web.
Clerk / OAuth HIPAA-scoped tokens Rate limiting
AI Agent Layer
Four specialized agents: Intake, Scheduling Copilot, Encounter Prep, Engagement & Continuity — each with scoped tool access and policy-gated autonomy.
RAG architecture Vector DB (Pinecone / Weaviate) LLM API (Claude / GPT) Structured outputs
Scheduling System
Provider availability calendar, booking/rescheduling engine, scheduling rules, and 6–12 month follow-up windows publication.
FHIR / REST GraphQL (org-dependent) Slot inventory
Event Bus + Orchestrator
Lifecycle event publishing, async coordination, idempotent execution, retry/DLQ handling for 15 defined appointment events.
Kafka / SNS / SQS Idempotency keys Dead-letter queues Correlation IDs
EHR / Provider Dashboard
Encounter status, clinical context, AI summary display, provider review workflow, visit completion trigger source.
FHIR / HL7 EHR API Provider-in-the-loop
Analytics & Reporting
Satisfaction trends, follow-up conversion, no-show/lateness metrics, and closed-loop scheduling optimization feedback.
ETL/ELT pipeline KPI dashboards Model feedback loops

Security & Compliance Controls
🔐
Encryption & Access
PHI encrypted in transit (TLS 1.3) and at rest. RBAC/ABAC with least-privilege access scoped per service. No PHI exposure to third-party tools without explicit need.
📝
Full Audit Logging
Every AI decision, tool call, and event carries a correlation ID and is logged for compliance. Compliance team audits sampled traces quarterly.
⚖️
Ethical AI Guardrails
Recommendation outcomes monitored across patient groups for fairness. Transparency views for patient/provider decisions. Opt-out and preference controls respected.
🚦
Human-in-the-Loop
Providers review AI summaries before clinical use. High-risk outputs escalated for manual review. Low-confidence AI actions suppressed and routed to staff queue.
🔔
Consent Management
Notification channel consent tracked per patient. Survey and follow-up workflows respect opt-in status. Data retention and deletion policies enforced at pipeline level.
🏛️
Governance Framework
Launch gates require product + clinical + compliance signoff. Alignment to Trustworthy AI principles and applicable healthcare privacy regulations. Continuous policy checks.
